What is an Operating System?

The ISO/IEC 2382 standard defines an operating system as “software that controls the execution of programs and that may provide services such as resource allocation, scheduling, input/output control, and data management.”. A deprecated German standard, the DIN 44300, defined an operating system as “the programs of a digital computer system that together with the properties of the computing system form the basis for the possible operating modes of the digital computing system, and which in particular control and monitor program execution.” Both definitions are pretty much the same, emphasising control and execution. The latter though, brings up the “properties of the computing system”. This reinforces that an operating system is hardware-dependent software. In turn, hardware is domain-dependent: a PC, a microwave, a fighter jet radar, a smart card… Traditional operating system textbooks define an OS as a “resource manager and an extended machine”. As a manager, it orchestrates resource allocation to potentially conflicting application requests; as an extended machine, it makes it easier to design applications by providing a set of abstractions and mechanisms that abstract the hardware.

1. The Kernel Abstraction

Pir Brinch Hansen coined the kernel concept in 1969 when designing the RC4000 operating system. At the time, he used the terms system monitor or nucleus. He aimed to provide a reusable, extensible mechanism to construct other operating systems around the same software core for the same machine. In his work he describes the role of a kernel (nucleus) in an operating system:

“The system nucleus coordinates multiprogramming and communication processes - a monitor with complete control of input/output, store protection, and interrupt response. I do not regard the monitor as an independent process but rather as a software extension of the hardware structure that makes the computer more attractive for multiprogramming. Its function is to implement the process concept and the primitives that processes can call to create and control other processes and communicate with them. It is the only program which executes privileged instructions.”

The description provided by Hansen remains. The traditional definition of an operating system as a resource manager and extended machine overlaps with the definition of a kernel. However, not every service an operating system provides must be at the trusted level. A microkernel has a small set of services running in privileged mode, often task scheduling, inter-task synchronisation, and inter-task communication. The remaining services are run on the application (unprivileged) level, each one within a dedicated task (or process). Application software and system software communicate by exchanging messages – in its simplest form, as a client-server model. Curiously, what Per Brinch Hansen described was, conceptually, a microkernel. A kernel is said to be monolithic when all operating system services (scheduling, inter-task synchronisation and communication, file system, drivers, etc.) run in privileged mode. On a simple architecture, the kernel runs as a single process; thus, all system services share the same address space. Any modularity from one service to another is at the logical level. A microkernel takes the Least Privilege Principle to the extreme—mechanisms should be given the minimum privilege necessary to perform their jobs. The computational cost of switching privilege levels and interprocess communication makes microkernels slower than monolithic kernels (for example, Windows NT was first launched as a microkernel and moved to a hybrid approach after performance bottlenecks).

2. Now we are talking…​

Formally, the “kernel” distinction can be made only if a hardware mechanism enables the separation of privileges for code and data. The takeaway is that when this hardware is absent, the kernel is a purely logical layer, and the operating system is a library-based or kernel-less operating system. In this case, the discussion of microkernel vs. monolithic kernel falls into the system’s logical architecture. A microkernel architecture uses message passing to communicate with the application layer. Typical kernel services run as isolated tasks. A monolithic kernel would, instead, have all its services within a single logical unit. That famous discussion between two (well-equipped) operating system designers, there’s a line worth mentioning, it is more or less like: "you say <OSName> is portable, but you could translate it as" it does not use any of the CPU features".

3.1. Kernel '0'

K0BA is like RC4000 with a strict embedded mindset. It implements a Task model and the primitives they use to control and communicate with other tasks. If no details are to be given, a 2-layer model fits. The upper layer (Layer 1), is regarded as the "Executive" or "Scheduler"; and the lower (Layer 0), is the "Dispatcher" or "Low-level Scheduler". This lower layer, is a software extension of the hardware, in the sense the next abstraction level is the ISA itself.

4. K0BA Scheduler

The K0BA scheduler is a priority-based, rate-monotonic scheduler. The most remarkable characteristic is having O(1) time complexity This means that, no matter the number of tasks to choose from, the scheduler will always spend the same amount of (logical) time.

Below, I detail how it was achieved.

4.1. Data Structures

The backbone of the queues where tasks will wait for its turn to run is a circular doubly linked list: removing any item from a doubly list takes O(1) (provided we don’t need to search the item). As the kernel is aware of each task’s address, adding and removing is always O(1). Singly linked lists, can’t achieve O(1) for removal in any case.

One design choice that does not change the time complexity but does improve cyclomatic complexity is having a DUMMY reference, which anchors a list and eases the handling of 'edge cases'.

Thus, comes another design choice towards achieving O(1). The global READY QUEUE is a table of FIFO queues—each queue dedicated to a priority—and not a single ordered queue. So, enqueuing a ready task is always O(1). If tasks were placed on a single READY QUEUE, the time complexity would be O(n), given the sorting needed. Yes, we pay with memory for efficiency, a typical trade-off, I might say.

So all this boils down to: enqueuing/dequeuing from any task queue on the system, global or associated with a kernel object, is always O(1). Now, it is up to the algorithm that picks the next task to run also to honor O(1).

4.2. The scheduling algorithm

It goes like this: as the READY QUEUE table is indexed by priority - the index 0 points to the queue of ready tasks with priority 0, and so forth, and there are 32 possible priorities - a 32-bit integer can represent the state of the ready queue table. It is a BITMAP:

The BITMAP computation: ((1a) OR (1b)) AND (2), s.t.:

(1a) Every Time a task is readied, update MASK performing: BITMAP |= (1U << task->priority );
(1b) Every Time an empty READY QUEUE becomes non-empty, update MASK by BITMAP |= (1U << queueIndex)
(2): Every Time READY QUEUE becomes empty, update MASK performing: BITMAP &= ~(1U << queueIndex);
MASK EXAMPLE:

    Ready Queue IDX :      6 5 4 3 2 1 0
          Not empty :      1 1 1 0 0 1 0
                           ------------->
                 (LOW)  Effective Priority  (HIGH)

In this case, the scenario is a system with 7 priority task levels. Queues with priorities 6, 5, 4, and 1 are not empty. All other READY QUEUES are empty, meaning these tasks are either SLEEPING, PENDING or BLOCKED. If we want to make any assumptions about what is happening, it could be the case that there was a single task with priority 0, that has just been suspended. Now, a task with priority 1 will get the joy of running…​

Given this mask, we know that we shall start inspecting on the LSBit and stop when the first 1 is found. There are uncountable manners of doing this. The approach I chose is:

(1) Isolate the rightmost '1':

`RBITMAP = BITMAP & -BITMAP. (- is the bitwise operator for two’s complement: ~BITMAP + 1) `

In this case:

                           [31]       [0]  :  Bit Position
                             0...1110010   :  BITMAP
                             1...0001110   : -BITMAP
                            =============
                             0...0000010   :  RBITMAP
                                     [1]

The rationale here is that, for a number N, its 2’s complement -N, flips all bits - except the rightmost '1' (by adding '1') . Then, N & -N results in a word with all 0-bits except for the less significant '1'.

(2) Extract _position of the rightmost '1':_

Within GCC, the _builtin_ctz() function does the trick: it returns the number of _trailing 0-bits within an integer, starting from the LSbit. The number of 'trailing zeroes' equals the position where the first '1' is found, that is also the ready queue index (and hence the priority) of the next task to be dispatched.

Using ARMv7M instructions, a possible solution is to use the CLZ (count lead zeros):

.global __getReadyPrio
.type __getReadyPrio, %function
.thumb_func
__getReadyPrio:
CLZ  R12, R0
MOV  R0, R12
NEG  R0, R0
ADD  R0, #31

BX LR

Thus, we subtract 31 from the number of leading zeroes, and get the index.

The source code in K0BA looks like:

5. Kernel Determinism

This is a simple test to establish some evidence the scheduler obeys the preemption criteria: a higher priority task always preempts a lower priority task.

Task1, 2, 3, 4 are in descending order of priority. If the scheduler is well-behaved, we shall see counters differing by "1".

volatile UINT32 counter1;
volatile UINT32 counter2;
volatile UINT32 counter3;
volatile UINT32 counter4;

VOID Task1(VOID)
{
	while(1)
	{
		counter1++;
		kPend();
	}
}

VOID Task2(VOID)
{
	while(1)
	{
		counter2++;
		kSignal(1); /* shall immediately be preempted by task1 */
		kPend();    /* suspends again */

	}
}


VOID Task3(VOID)
{
	while(1)
	{
		counter3++;
		kSignal(2);  /* shall immediately be preempted by task2 */
		kPend();     /* suspends again */
	}


}

VOID Task4(VOID)
{
	while(1)
	{
	    counter4++;
	    kSignal(3); /* shall immediately be preempted by task3 */
                        /* as the lowest priority task it will only be resumed
                           after all higher priority tasks are suspended */
	}

}

6.1. Direct Task Signal

  Some tasks are entirely reactive and have a single responsibility. It is cheaper and more effective to allow them to pend themselves - and be signalled
  without an associated kernel object - such as a semaphore. The `signal()` primitive takes a Task ID as a  parameter. The `pend()` primitive takes no parameters, acts on the caller task. If a task is signalled when not pending, the error counter "lostSignals" is
  increased in its task control block. This counter is meant for some diagnostics. The standard mechanism does not check for lost signals - to catch up with. Doing so would turn it into a (degenerated) "weak" semaphore (explained later).
 A `PENDING` task is placed on a global sleeping queue and removed when signalled—its position in the queue does not matter.
The main use case is for deferred handlers.
The lack of a control block associated with a direct signal makes it rudimentary but sufficient, justified by the negligible cost and the ubiquity of single-purpose event-driven tasks sitting idle until notified: after readied, the scheduler will take care of its dispatching, respecting the priority. No shared resource, no contention, no priority inversion.

There are two other direct signal-like primitives: suspend(task id) and resume(task id). With them, a task might pause another task to resume it later.

Direct signals cannot be disabled - no cost, no saving.

6.2. Semaphores

In K0BA semaphores can be classified as 'strong counter' semaphores. Counter means the primitives signal() and wait() will increase and decrease, respectively, the initial signed value 'N' a semaphore is assigned (a semaphore cannot be initialised with a negative value). When wait() results in a negative value, the caller will be blocked within the semaphore queue. The negative value of a counter semaphore inform us straight away how many tasks are blocked waiting for a signal. When signalled only one task is unblocked. If a semaphore has a dedicated queue it is 'strong', because it establishes an order. Not having a queue, make a 'weak' semaphore, as the task to be released depends on the position of the task wh who signals. In K0BA, tasks block and resume within a semaphore-guarded region, ordered by their priority. Semaphores implement a priority inheritance protocol to diminish priority inversion.

Semaphores service can be enabled/disabled, but the Message Queue service currently depends on them.

6.3. Mutexes

Mutexes are 'locks'. A locked resource has an owner, and only the owner can release it—a task blocks on a locked mutex.
Mutexes also have a priority inversion protocol. The owner's priority is boosted if a higher priority task blocks trying to access (lock) its guarded region.  When a task tries to unlock a mutex it does not own, the behaviour is implementation-dependent.. In K0BA, trying to unlock an owned mutex leads to an (elegant) disgraceful failure.

About mutexes vs sempahores: If you initialise a semaphore as 1, and GUARANTEE that this semaphore won’t be touched by who is not trying to enter/leave the critical region, the effect of mutual-exclusion is the same. A mutex guarantees ownership, and thus, accidental unlocking will not happen. The thing is that if your system has code poking kernel objects it shouldn’t - it has a logical synchronisation problem, and most certainly will fail. Some RTOSes with dynamic task creation can also use the owner identity to forbid a task that owns a mutex to be terminated or to identify its termination: the action if it happens, is implementation-defined. Recursive locking is also only possible because a mutex has an owner. These characteristics are irrelevant within K0BA, as dynamic allocation is meant to be an exception).

Mutex service can be enabled/disabled.

6.4. Events

An event is a condition in time that will trigger a reaction: a countdown timer reaching zero, or the 7th bit of the 7th received stream on the 7th pin in the 7th fullmoon night in the 7th year of system uptime - being 0. These are 2 events. The reactions we don’t bother, it can even be 'to ignore the event'. Within K0BA, an event is associated to a kernel object. The kernel assigns only an ID and a sleeping queue within an' event' object. Here, an event is a 'pure signal'—it is present or absent. A semaphore is a record of pure signals (and actually, that was the critical insight that led to the (brilliant) semaphore idea). What is the difference, then? Say a modern car has its speed increased by the driver. Many non-safety-related mechanisms are warned about it, like the radio so that it might boost the volume. The windshield wiper control will increase its speed if they are on. Not to mention the speed alarm. The cruiser being turned off might not be within the same group as the radio (I hope). One event might be of interest to many devices - a semaphore can’t do that - for each interested reader, it would have to signal once. Notably, a queue would not be necessary since all tasks are woken by a wake(event). It is employed, though, so the wake() primitive does not need to traverse the list of sleeping tasks, catching each one that matches a specific event. Given that all tasks are made READY, if a woken task has a higher priority than the running task, it takes CPU over.

So, we have direct signals, semaphores, mutex and sleep-wake. And actually, they overlap in the sense they all handle events, but how they handle is enough to justify each of them as a different service:

  • The direct signals saved us the overhead of a semaphore structure and operations for a prevalent task pattern within embedded system software: the pend-does-one-thing-pend, such as interrupt service routines split in half-bottom and half-top.

  • The semaphore accumulates 'ups' and 'downs' of a signal - an increment/decrement + a test within an atomic operation. In enforces them a precedence on acquiring whatever the semaphore is guarding. If the semaphore counter is initialised as '1', effectively, it allows a single running unit to be within the region it is guarding.

  • The mutex provides the ownership property for a '1'-semaphore, making it easier to track errors.

  • The sleep-wake provides a broadcast signal for events we don’t bother to track its history.

Sleep/Wake Events service are optional and can be turned ON/OFF in kconfig.h.

6.5. Remarks on synch services

6.5.1. About event flags and event groups

Sometimes we are interested in the combination of the absence/presence of several events on a given interval - so, we need a storage. RTOSes often provide a service for that: 'event flags' within 'event groups'. The subscribers sleep after a get() with the values they expect. The publisher set(). The broker reads, evaluates and wake() all interested readers. As seen, it describes a public mailbox storing an integer (with flags). You will need more data structures, usually as simple as a table. But how the execution units handle this data - on my perspective - does not justify new abstractions. It means more code, bugs, maintenance, and another API the kernel designer makes clear—another the user needs to learn. (Hoare usually discussed the delicate balance that exists between specialisation and generalisation. Hansen would argue that opaque complexity within a concurrent environment will worsen things.)

6.6. About blocking primitives time-out

It is common for RTOSes to have a time-out the user assigns when calling a sleeping primitive. I wonder the right way to use this feature. What does a task do after this time-out? If a task tries to access a critical region and blocks forever, it is probably a system failure. We are better off catching it and fixing it up. As I see, the best approach would be to sleep on a timer (sleep(ticks)), wake up and test if the condition is met - if not, branch for a clean-up/recovery or even a reset. You might also test the condition before locking. Anyway, for semaphores, this feature is available - you use the attempt (s, n) primitive, providing the number of TICKS before giving up trying to acquire a semaphore. In this case the task will enter a special state called ATTEMPTING. After N ticks have elapsed, if it can’t acquire the resource/region, the call returns an error code. The user might check for this return so the task can branch to an alternative path. You can use a query() to get the current value of the counter semaphore before waiting. For mutexes, you can query its lock state; for events, you can query the number of tasks within its sleeping queue. Finally, for this usage pattern—checking before blocking—the tailored primitive within K0BA is an asynchronous mailbox: there is no kernel-imposed context-switching for a retry.

6.7. About "signals" installed callbacks

It is tempting to become UNIX-minded and install callbacks on signals. When resuming, a task first checks for any pending signals. If they exist, the task deviates from its normal flow (as within any software interrupt), runs the callback, and deals with any side effects. The overhead is excessive—both in time and space—a need to reserve a space on the stack or provide a stack for the callback; a need to tune the context-switching to happen when there is a signal pending and to get back to the deviated point. Since it kills any determinism, it is out of the kernel’s concerns—despite real-time gospel says they are a big no; as I am not up for any gospel, I’m just providing a guideline. The application timers have installed callbacks (as they are periodic, the kernel can handle them).

7. Inter-task communication services

7.1. Mailboxes

In K0BA, a mailbox is a kernel object whose capacity is 'one message'. You write to a mailbox, then it is full. You read, it is empty. A mailbox stores a 32-bit address. The difference between a mailbox and a regular message queue (and these terms are often interchangeable) is that a message queue can be depicted as a circular data structure with a mailbox on the head and another on the tail. When both are full/empty, the queue will be logically full/empty.

Blocking/synchronous mailboxes fit well when tasks NEED to synchronise, so-called "rendez-vous". If using the ACK option - sender waits for a (kind of) direct signal to be sure the receiver got the message. Thus, with ACK, a mailbox is not only blocking - but synchronous - an extended rendez-vous. In this case they lock-step, a 'tick-bump-tick-bump…​' - logically there is no buffering, as buffering and logical time are tied - and the mailbox is considered with 'zero-capacity'. It does not create (more) contention because to wait for the ACK, the the writer frees the mailbox by setting it full.

Beware: It is an indirect mailbox, so we cannot really guarantee who receives the message. Thus, the ACK extension does not make sense if the sender only wants to know if some task read its message—it is enough not to block on a full mailbox again to know some task received its message: it is for lock-stepping tasks with no buffering.

Importantly, synch mailboxes are binary semaphores in their own right—they initialise with a state (full/empty), have a built-in priority inheritance mechanism and have a waiting queue. You might use them as binary semaphores. An overkill? If you have semaphores or mutexes enabled, yes—a synch mailbox is more than double a semaphore size (couldn’t be different; think about the producer-consumer problem solved with semaphores: it takes a semaphore for items, a semaphore for space, and a semaphore for mutual exclusion). I have been writing apps with K0BA using only mailboxes and sleep-wakes (all other services disabled).

When tasks block on a mailbox, they are released by priority.

Asynchronous mailboxes don’t have built-in blocking mechanisms and rely on the value of the mail address attached to them: if NULL, it is empty. A regular send() primitive to write on an async mbox will only be successful if the box is empty (NULL mail) and a recv(), will copy the contents of the mail address to a buffer the consumer passed as a parameter. The mailbox will 'destroy' the message, so it is empty again. A recvkeep() primitive does not destroy the message, and a sendovw() does not care if the box is full or not - it overwrites.

When combined with synchronisation services, you can create flexible communication channels.

For instance, you can create a BROADCAST communication channel. Say, an ISR deposits a message from a peripheral on an asynch mbox, and wakes all tasks sleeping on an event associated to the mailbox and returns. The woken tasks use recvkeep() - thus every task will get a copy of the message. (Within an ISR we cannot use blocking primitives).

I regard mailboxes as powerful building blocks for task management, particularly in their asynchronous and synchronous extremes.

Both synch mailboxes and asynch mailboxes are optional and can be turned ON/OFF in kconfig.h.

8. Blocking Message Queue

The message queue service can carry messages up to 255 bytes. It works with an allocator. Given the circular FIFO nature, a recv() primitive copies the message to an address the receiver provides, so the queue buffer can be reused without corrupting data.

You can choose to not copy messages but send and receive pointers. In this case, the memory you statically allocated for the queue is not passed as a parameter during the initialisation, you pass it as NULL, and use the sendptr() and recvptr(). The way you manage you memory is up to you, but, I recommend using the allocator service. You send the message address and guarantees it will be in scope when the receiver gets it - the message queue service will not touch this pool. The receiver might it self after calling `recvptr()`free the memory. This method will work particularly when you do not need a deepy copy, that is, the data does not points to other pointers.

A message queue will behave asynchronously until a receiver try to read from an empty queue, or a producer tries to write on a full queue. So, they will block. This behaviour might be desirable or not. The result, anyway, is that tasks with different periods end up synchronising to the lowest rate task - (usually, undesirable for real-time). After a queue blocks full, when it eventually unblocks you might had lost data. Is it the problem? Yes and no. The problem with a queue is that by buffering you are reading from the past. Between a message that does not arrive, and one that arrives with information that is not useful, there is no practical difference. But if a message arrives with information that will deceive your control loop to perform a wrong action - this is the worse. That can be avoided if your send-receive pair follows an unidirectional order, and handle predictable timing events (loosely, periodic events). Finally, the queue size must be adequate (look at Little’s theorem). So, except for logging/monitoring or serialize access, queues might cause trouble when a system takes action with the information it is reading.

Important to note that although the Message Queue API asks for the address for the message contents, there is an internal pool of 'messages records': they wrap the message address with the sender ID and the message size, and that is what is transmitted/received. Then, a message queue internally is a list of message records (…​ this was done so you can send messages up to a maximum size within the same queue, and identifying the sender ID is often useful). The number of message records is defined in kconfig.h Also, you can make the send() primitive non-blocking, so it will overwrite the current message on the tail if the queue is full (kconfig.h). Blocking send() and recv() releases blocked tasks by priority.

(I personally rarely use service queues when working with 3rd party rtoses. A linear/circular buffer or a list + semaphores will do the job with no overkill - if it is shared memory. Consider that before using this service.)

(Interesting: in UNIX a broken pipe is a pipe with writers but no readers. A pipe with readers but no writers is not 'broken'. It reinforces the idea that there is not such a thing as valid never-received data.)

Message queues are optional and can be turned ON/OFF in kconfig.h. You cannot receive from a queue within ISRs. To send from an ISR, it shall be non-blocking.

8.1. Asynch Pump-Drop Message Queue

These are queues meant to overcome the drawbacks exposed above for message queues, based on the concept of Cyclical Asynchronous Buffers (CAB) (essentialy as found in the HARTIK kernel). These queues are a one-to-many mechanism. A buffer can be read by several consumers, but no buffer can be written while consumed. The ideal queue size is the number of readers + 1.

The semantics is as follows:

  • When the producer needs to write a new message to the queue, first it reserves a buffer. With a valid address, it attaches to the buffer the mesage address and size, and 'pump' it to the queue. Now, this message is available to be read by several consumers - until it is overwritten.

  • The last message pumped in a queue is not overwritten UNTIL a new buffer is pumped and finds its number of clients is zero (all readers dropped)

  • A reader first 'fetches' a message buffer address - what will increase the user count within a buffer. After consuming the message, the reader 'drop the buffer'.

By the exposure, you can conclude that a reader might miss messages when the writer’s rate is higher. A reader within a (servo-loop) controller can afford a 'miss' rate— what leads to failure is reading messages that do not represent the plant state.

Pump-drop queues are optional and can be turned ON/OFF in kconfig.h.

8.2. Pipes

Pipes transmit streams of bytes - unlike queues or mailboxes that transmit 'objects' with a format. They are fast because data does not move, what moves is the read/write pointer within the pipe buffer. A writer stops its operation when the N specified bytes have been V or there is no more room. In the last case it goes to sleep until a reader wakes it up after 'extracting' (moving the read pointer) from the pipe.

A reader reads until N specified bytes were read or when there is no more data. In the last case it will sleep, to be waken by a writer. Both one-to-many and many-to-one channels are possible - those who wake need to themselves ensure precedence.

A many-to-many channel is possible, but harder to synchronise. Consider employing simple circular buffers following the producer-consumer solution before using this service.
