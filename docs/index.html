It is a one-man, unfunded, unprofittable work. It can be classified as a by-product of the author’s research on the subject, a laboratory, a manifesto or testament of being ''fueled by the craft''</p> </div> <div class="paragraph data-line-13"> <p>The "zero" represents a low-overhead design based on "grounded" system design concepts. While aiming for modularity, it is essential to avoid opacity. This approach stems from my lessons while pursuing ADT within embedded real-time systems.</p> </div> <div class="paragraph data-line-15"> <p>From what the user regards as the <em>application</em> down to the lowest kernel layer - at most three layers exist - <em>executive</em>, <em>inter-task management</em> and <em>memory management</em>.</p> </div> <div class="imageblock data-line-18"> <div class="content"> <img src="images/layeredkernel.png" alt="K0BA architecture"> </div> </div> <div class="paragraph data-line-19"> <p><em>Figure 1. K0BA layered architecture - the dashed line within the API reminds the 'application' layer is an illusion</em></p> </div> <hr> <div class="paragraph data-line-25"> <p>“<em>I conclude that there are two ways of constructing a software design: One way is to make it so simple that there are obviously no deficiencies and the other way is to make it so complicated that there are no obvious deficiencies. The first method is far more difficult. It demands the same skill, devotion, insight, and even inspiration as the discovery of the simple physical laws which underlie the complex phenomena of nature. It also requires a willingness to accept objectives which are limited by physical, logical, and technological constraints, and to accept a compromise when conflicting objectives cannot be met. (Tony Hoare, 1980)</em>”</p> </div> <hr> </div> </div> <div class="sect1 data-line-29"> <h2 id="operating_systems_and_the_kernel_abstraction">1. Operating systems and the 'Kernel' abstraction</h2> <div class="sectionbody"> <div class="paragraph data-line-31"> <p>The ISO/IEC 2382 standard defines an operating system as “software that controls the execution of programs and that may provide services such as resource allocation, scheduling, input/output control, and data management.”.</p> </div> <div class="paragraph data-line-33"> <p>A deprecated German standard, the DIN 44300, defined an operating system as “the programs of a digital computer system that together with the properties of the computing system form the basis for the possible operating modes of the digital computing system, and which in particular control and monitor program execution.” Both definitions are pretty much the same, emphasising control and execution. The latter though, brings up the “properties of the computing system”. This reinforces that an operating system is hardware-dependent software. In turn, hardware is domain-dependent: a PC, a microwave, a fighter jet radar, a smart card…</p> </div> <div class="paragraph data-line-35"> <p>Traditional operating system textbooks define an OS as a “resource manager and an extended machine”. As a manager, it orchestrates resource allocation to potentially conflicting application requests; as an extended machine, it makes it easier to design applications by providing a set of abstractions and mechanisms that abstract the hardware.</p> </div> <div class="sect2 data-line-38"> <h3 id="the_roots">1.1. The roots</h3> <div class="paragraph data-line-40"> <p>Pir Brinch Hansen coined the kernel concept in 1969 when designing the RC4000 operating system. At the time, he used the terms <em>system monitor</em> or <em>nucleus</em>. He aimed to provide a reusable, extensible mechanism to construct other operating systems around the same software core for the same machine. In his work he describes the role of a kernel (nucleus) in an operating system:</p> </div> <div class="paragraph data-line-42"> <p>“<em>The system nucleus coordinates multiprogramming and communication processes - a monitor with complete control of input/output, store protection, and interrupt response. I do not regard the monitor as an independent process but rather as a software extension of the hardware structure that makes the computer more attractive for multiprogramming. Its function is to implement the process concept and the primitives that processes can call to create and control other processes and communicate with them. It is the only program which executes privileged instructions.</em>”</p> </div> <div class="imageblock data-line-46"> <div class="content"> <img src="images/kernelsuperuser.png" alt="kernelsuperuser"> </div> </div> <div class="paragraph data-line-48"> <p>The description provided by Hansen remains. The traditional definition of an operating system as a resource manager and extended machine overlaps with the definition of a kernel. However, not every service an operating system provides must be at the trusted level.</p> </div> <div class="paragraph data-line-50"> <p>A microkernel has a small set of services running in privileged mode, often task scheduling, inter-task synchronisation, and inter-task communication. The remaining services are run on the application (unprivileged) level, each one within a dedicated task (or process). Application software and system software communicate by exchanging messages – in its simplest form, as a client-server model. Curiously, what Per Brinch Hansen described was, conceptually, a microkernel.</p> </div> <div class="paragraph data-line-52"> <p>A kernel is said to be monolithic when all operating system services (scheduling, inter-task synchronisation and communication, file system, drivers, etc.) run in privileged mode. On a simple architecture, the kernel runs as a single process; thus, all system services share the same address space. Any modularity from one service to another is at the logical level.</p> </div> <div class="paragraph data-line-54"> <p>A microkernel takes the Least Privilege Principle to the extreme—mechanisms should be given the minimum privilege necessary to perform their jobs. The computational cost of switching privilege levels and interprocess communication makes microkernels slower than monolithic kernels (for example, Windows NT was first launched as a microkernel and moved to a hybrid approach after performance bottlenecks).</p> </div> </div> <div class="sect2 data-line-56"> <h3 id="now_we_are_talking">1.2. Now we are talking</h3> <div class="paragraph data-line-58"> <p>Formally, the “kernel” distinction can be made only if a hardware mechanism enables the separation of privileges for code and data. The takeaway is that when this hardware is absent, the kernel is a purely logical layer, and the operating system is a library-based or kernel-less operating system. In this case, the discussion of microkernel vs. monolithic kernel falls into the system’s logical architecture.</p> </div> <div class="paragraph data-line-60"> <p>A microkernel architecture uses message passing to communicate with the application layer. Typical kernel services run as isolated tasks. A monolithic kernel would, instead, have all its services within a single logical unit.</p> </div> <div class="paragraph data-line-62"> <p>In that infamous discussion between two (general-purpose) operating system designers, there’s a line more or less like: "you say <OSName> is portable, but you could translate it as" it does not use any of the CPU features". This statement shall make anyone working with low-level software to wonder.</p> </div> <div class="paragraph data-line-64"> <p><strong>Mission: embedded</strong></p> </div> <div class="paragraph data-line-66"> <p>With no memory virtualisation, and no affordable swapping, what is left is a single program image and a single resident process. We tweak it to create Threads. Threads are semi-independent (clueless) running units - there is no isolation between a thread and another. Yes, you can have an MPU (K0BA has a version using it) - still - it is not virtualisation, as you see on <em>multiprogramming</em> kernels. They are memory firewalls - except that this firewall burns the house down when violated.</p> </div> <div class="paragraph data-line-69"> <p>K0BA refuses the idea that we can afford an <em>application layer</em> that is <em>hardware-agnostic</em> within constrained embedded systems. In this realm, application software meets system software, and system software morphs into hardware-dependent software before we know it.</p> </div> <div class="paragraph data-line-71"> <p>Besides, real-time means our time domain needs to tick at the same pace of the changes in the environment we control. This alone is enough to avoid indirection levels the most we can. Abstractions - generalisations or specialisations - will always work with indirections. Here I shall make an extrapolation to be understood: take for instance, interpreted languages. On a nutshell, they are hash-tables of "virtual instances" assigned to concrete implementations. All the control and prediction we care is gone within heavy abstractions.</p> </div> <div class="paragraph data-line-73"> <p>Truly independent applications layers rely on a level of abstraction <em>we cannot afford</em>. Denying this fact is harmful. Overlooking it is counterproductive. We shall embrace it.</p> </div> <div class="paragraph data-line-75"> <p><strong>The Kernel '0'</strong></p> </div> <div class="paragraph data-line-77"> <p>K0BA is like RC4000 with a <em>strict embedded</em> mindset. It implements a Task model and the primitives they use to control and communicate with other tasks. If no details are to be given, a 2-layer model fits. The upper layer (Layer 1), is regarded as the "Executive" or "Scheduler"; and the lower (Layer 0), is the "Dispatcher" or "Low-level Scheduler". This lower layer, is a software extension of the hardware, in the sense the next abstraction level is the ISA itself.</p> </div> </div> </div> </div> <div class="sect1 data-line-80"> <h2 id="k0ba_scheduler">2. K0BA Scheduler</h2> <div class="sectionbody"> <div class="paragraph data-line-82"> <p>The K0BA scheduler is a priority-based, rate-monotonic scheduler. The most remarkable characteristic is having O(1) time complexity This means that, no matter the number of tasks to choose from, the scheduler will always spend the same amount of (<code>logical</code>) time.</p> </div> <div class="paragraph data-line-85"> <p>Below, I detail how it was achieved.</p> </div> <div class="sect2 data-line-87"> <h3 id="data_structures">2.1. Data Structures</h3> <div class="paragraph data-line-89"> <p>The backbone of the queues where tasks will wait for its turn to run is a circular doubly linked list: removing any item from a doubly list takes O(1) (provided we don’t need to search the item). As the kernel is aware of each task’s address, adding and removing is always O(1). Singly linked lists, can’t achieve O(1) for removal in any case.</p> </div> <div class="paragraph data-line-91"> <p>One design choice that does not change the time complexity but does improve cyclomatic complexity is having a DUMMY reference, which anchors a list and eases the handling of 'edge cases'.</p> </div> <div class="literalblock data-line-97"> <div class="content"> <pre> HEAD TAIL | |-->| |-->| |-->| |--> <--| | |DUMMY| | H | | H+1 | | H+2 | . . . | T | <-||<--||<--||<--|| ||--> ||</pre> </div> </div> <div class="ulist data-line-105"> <ul> <li class="data-line-105"> <p>INITIALISE</p> <div class="literalblock data-line-107"> <div class="content"> <pre>The list is initialised by declaring a node, and assigning its previous and next pointers to itself. The global READY QUEUE is a table of FIFO queues—each queue dedicated to a priority—and not a single ordered queue. So, enqueuing a ready task is always O(1). If tasks were placed on a single READY QUEUE, the time complexity would be O(n), given the sorting needed. Yes, we pay with memory for efficiency, a typical trade-off, I might say.</p> </div> <div class="listingblock data-line-150"> <div class="content"> <pre class="highlightjs highlight"><code class="language-none hljs"> READY QUEUE | IDX (Priority)

</div>
<div class="paragraph data-line-165">
<p>So all this boils down to: enqueuing/dequeuing from any task queue on the system, global or associated with a kernel object, is always O(1). Now, it is up to the algorithm that picks the next task to run also to honor O(1).</p>
</div>
</div>
<div class="sect2 data-line-168">
<h3 id="the_scheduling_algorithm">2.2. The scheduling algorithm</h3>
<div class="paragraph data-line-170">
<p>It goes like this: as the READY QUEUE table is indexed by priority - the index 0 points to the queue of ready tasks with priority 0, and so forth, and there are 32 possible priorities - a 32-bit integer can represent the state of the ready queue table. It is a BITMAP:</p>
</div>
<div class="listingblock data-line-172">
<div class="content">
<pre class="highlightjs highlight"><code class="language-none hljs">The BITMAP computation: ((1a) OR (1b)) AND (2), s.t.:

(1a) Every Time a task is readied, update: BITMAP |= (1U &lt;&lt; task-&gt;priority );
(1b) Every Time an empty READY QUEUE becomes non-empty, update: BITMAP |= (1U &lt;&lt; queueIndex)
(2): Every Time READY QUEUE becomes empty, update: BITMAP &amp;= ~(1U &lt;&lt; queueIndex);</code></pre>
</div>
</div>
<div class="listingblock data-line-180">
<div class="content">
<pre class="highlightjs highlight"><code class="language-none hljs">EXAMPLE:

  Ready Queue Index :     (6)5 4 3 2 1 0
          Not empty :      1 1 1 0 0 1 0
                           -------------&gt;
                 (LOW)  Effective Priority  (HIGH)</code></pre>
</div>
</div>
<div class="paragraph data-line-190">
<p>In this case, the scenario is a system with 7 priority task levels. Queues with priorities 6, 5, 4, and 1 are <em>not empty</em>.</p>
</div>
<div class="paragraph data-line-192">
<p>The idle task priority is assigned by the kernel, during initialisation taking into account all priorities the user has defined.  Unless user-tasks are occupying all 32 piorities, the Idle Task is treated as an ordinary lowest-priority and has a position in the queue. If not, the idle task on practice will have no queue position, and will be readied when the BITMAP is  <code>0</code>.
In the above bitmap, the idletask is in <code>readyQueue[6]</code> .</p>
</div>
<div class="paragraph data-line-195">
<p>All other READY QUEUES are empty, meaning these tasks are either SLEEPING, PENDING or BLOCKED.
If we want to make any assumptions about what is happening, it could be the case that there was a single task with priority 0, that has just been suspended. Now, a task with priority 1 will get the joy of running&#8230;&#8203;</p>
</div>
<div class="paragraph data-line-198">
<p>Given this mask, we know that we shall start inspecting on the LSBit and stop when the first 1 is found. There are uncountable manners of doing this. The approach I chose is:</p>
</div>
<div class="paragraph data-line-200">
<p>(1) <em>Isolate the rightmost '1':</em></p>
</div>
<div class="paragraph data-line-202">
<p>`RBITMAP = BITMAP &amp; -BITMAP. (- is the bitwise operator for two&#8217;s complement: ~BITMAP + 1) `</p>
</div>
<div class="paragraph data-line-204">
<p>In this case:</p>
</div>
<div class="listingblock data-line-206">
<div class="content">
<pre class="highlightjs highlight"><code class="language-none hljs">                           [31]       [0]  :  Bit Position
                             0...1110010   :  BITMAP
                             1...0001110   : -BITMAP
                            =============
                             0...0000010   :  RBITMAP
                                     [1]</code></pre>
</div>
</div>
<div class="paragraph data-line-214">
<p><em>The rationale here is that, for a number N, its 2&#8217;s complement -N, flips all bits - except the rightmost '1' (by adding '1') . Then, N &amp; -N results in a word with all 0-bits except for the less significant '1'.</em></p>
</div>
<div class="paragraph data-line-216">
<p>(2) <em>Extract <strong>_position</em></strong> of the rightmost '1':_</p>
</div>
<div class="paragraph data-line-218">
<p>Within GCC, the  <code><em>_builtin_ctz()</code> function does the trick: it returns the number of _trailing</em> 0-bits within an integer, starting from the LSbit. The number of 'trailing zeroes' equals the position where the first '1' is found, that is also the ready queue index (and hence the priority) of the next task to be dispatched.</p>
</div>
<div class="paragraph data-line-220">
<p>Using ARMv7M instructions, a possible solution is to use the CLZ (count lead zeros):</p>
</div>
<div class="listingblock data-line-222">
<div class="content">
<pre class="highlightjs highlight"><code class="language-none hljs">.global __getReadyPrio
.type __getReadyPrio, %function
.thumb_func
__getReadyPrio:
CLZ  R12, R0
MOV  R0, R12
NEG  R0, R0
ADD  R0, #31

BX LR</code></pre>
</div>
</div>
<div class="paragraph data-line-235">
<p>Thus, we subtract 31 from the number of leading zeroes, and get the index.</p>
</div>
<div class="paragraph data-line-238">
<p>The source code in K0BA looks like:</p>
</div>
<div class="listingblock data-line-240">
<div class="content">

<div class="sect2 data-line-406">
<h3 id="direct_task_signal">4.1. Direct Task Signal</h3>
<div class="paragraph data-line-408">
<p>Some tasks are entirely reactive and have a single responsibility. It is cheaper and more effective to allow them to pend themselves - and be signalled
without an associated kernel object - such as a semaphore. The <code>signal(id)</code> primitive takes a Task ID as a  parameter. The <code>pend()</code> primitive takes no parameters, acts on the caller task. If a task is signalled when not pending, the error counter "lostSignals" is
increased in its task control block. This counter is meant for some diagnostics. The standard mechanism does not check for lost signals - to catch up with. Doing so would turn it into a (degenerated) "weak" semaphore (explained later).
 A <code>PENDING</code> task is placed on a global sleeping queue and removed when signalled—its position in the queue does not matter.</p>
</div>
<div class="paragraph data-line-413">
<p>The main use case is for deferred handlers.
The lack of a control block associated with a direct signal makes it rudimentary but sufficient, justified by the negligible cost and the ubiquity of single-purpose event-driven tasks sitting idle until notified: after readied, the scheduler will take care of its dispatching, respecting the priority. No shared resource, no contention, no priority inversion.</p>
</div>
<div class="paragraph data-line-416">
<p>The is also the primitive <code>suspend(task id)</code>. In this case, a task suspends another. The suspended task switches to the state <code>SUSPENDED</code>. To resume it, the same <code>signal(task id)</code> primitive is used.</p>
</div>
<div class="admonitionblock warning data-line-420">
<table>
<tr>
<td class="icon">
<i class="fa icon-warning" title="Warning"></i>
</td>
<td class="content">
<div class="paragraph data-line-421">
<p>Using different tasks to suspend/signal another is a strong indication of misuse</p>
</div>
</td>
</tr>
</table>
</div>
<div class="paragraph data-line-424">
<p>Direct signals cannot be disabled - no cost, no saving.</p>
</div>
</div>
<div class="sect2 data-line-426">
<h3 id="semaphores">4.2. Semaphores</h3>
<div class="paragraph data-line-428">
<p>In K0BA semaphores can be classified as 'strong counter' semaphores. Counter means the primitives <code>signal()</code> and <code>wait()</code> will increase and
  decrease, respectively, the initial signed value 'N' a semaphore is assigned (a semaphore cannot be initialised with a negative value).
  When  <code>wait()</code> results in a negative value, the caller will be blocked within the semaphore queue. The negative value of a counter semaphore inform us straight away how many tasks are blocked waiting for a signal. When signalled only one task is unblocked. If a semaphore has a dedicated queue it is 'strong', because it establishes an order. Not having a queue, make a 'weak' semaphore, as the task to be released depends on the position of the task wh who signals.
  In K0BA, tasks block and resume within a semaphore-guarded region, ordered by their priority. Semaphores implement a priority inheritance protocol to diminish priority inversion.</p>
</div>
<div class="paragraph data-line-433">
<p>Semaphores service can be enabled/disabled, but the Message Queue service currently depends on them.</p>
</div>
</div>
<div class="sect2 data-line-436">
<h3 id="mutexes">4.3. Mutexes</h3>
<div class="paragraph data-line-437">
<p>Mutexes are 'locks'. A locked resource has an owner, and only the owner can release it—a task blocks on a locked mutex.
  Mutexes also have a priority inversion protocol. The owner&#8217;s priority is boosted if a higher priority task blocks trying to access (lock) its guarded region.  When a task tries to unlock a mutex it does not own, the behaviour is implementation-dependent.. In K0BA, trying to unlock an owned mutex leads to an (elegant) disgraceful failure.</p>
</div>
<div class="paragraph data-line-440">
<p><em>About mutexes vs semaphores</em>: If you initialise a semaphore as 1, and GUARANTEE that this semaphore won&#8217;t be touched by who is not trying to enter/leave the critical region, the effect of mutual-exclusion is the same. A mutex guarantees ownership,  and thus, accidental unlocking will not happen. The thing is that if your system has code poking kernel objects it  shouldn&#8217;t - it has a logical synchronisation problem, and most certainly will fail. Some RTOSes with dynamic task creation can also use the owner identity to forbid a task that owns a mutex to be terminated or to identify its termination: the action if it happens, is implementation-defined. Recursive locking is also only possible because a mutex has an owner. These characteristics are irrelevant within K0BA, as dynamic allocation is meant to be an exception).</p>
</div>
<div class="paragraph data-line-442">
<p>Mutex service can be enabled/disabled.</p>
</div>
</div>
<div class="sect2 data-line-444">
<h3 id="events">4.4. Events</h3>
<div class="paragraph data-line-446">
<p>An event is a condition in time that will trigger a reaction: a countdown timer reaching zero, or the 7th bit of the 7th received
  stream on the 7th pin in the 7th fullmoon night in the 7th year of system
  uptime - being 0. These are 2 events. The reactions we don&#8217;t bother, it can even be 'to ignore the event'.
  Within K0BA, an event is associated to a kernel object. The kernel assigns only an ID and a sleeping queue within an' event' object. Here, an event is a 'pure signal'—it <em>is</em> present or absent. A semaphore is a <em>record</em> of pure signals  (and actually, that was the critical insight that led to the (brilliant) semaphore idea). What is the difference, then?
Say a modern car has its speed increased by the driver. Many non-safety-related mechanisms are warned about it, like the radio so that it might boost the volume. The windshield wiper control will increase its speed if they are on. Not to mention the speed alarm. The cruiser being turned off might not be within the same group as the radio (I hope). One event might be of interest to many devices - a semaphore can&#8217;t do that - for each interested reader, it would have to signal once.
Notably, a queue would not be necessary since all tasks are woken by a <code>wake(event)</code>. It is employed, though, so the <code>wake()</code> primitive does not need to traverse the list of sleeping tasks, catching each one that matches a specific event. Given that all tasks are made READY, if a woken task has a higher priority than the running task, it takes CPU over.</p>
</div>
<div class="paragraph data-line-453">
<p>So, we have direct signals, semaphores, mutex and sleep-wake. And actually, they overlap in the sense they all handle events, but <em>how</em> they handle is enough to justify each of them as a different service:</p>
</div>
<div class="ulist data-line-455">
<ul>
<li class="data-line-455">
<p>The direct signals saved us the overhead of a semaphore structure and operations for a prevalent task pattern within embedded system software: the pend-does-one-thing-pend, such as interrupt service routines split in half-bottom and half-top.</p>
</li>
<li class="data-line-456">
<p>The semaphore accumulates 'ups' and 'downs' of a signal - an increment/decrement + a test within an atomic operation. In enforces them a precedence on acquiring whatever the semaphore is guarding. If the semaphore counter is initialised as '1', effectively, it allows a single running unit to be within the region it is guarding.</p>
</li>
<li class="data-line-457">
<p>The mutex provides the ownership property for a '1'-semaphore, making it easier to track errors.</p>
</li>
<li class="data-line-458">
<p>The sleep-wake provides a broadcast signal for events we don&#8217;t bother to track its history.</p>
</li>
</ul>
</div>
<div class="paragraph data-line-461">
<p>Sleep/Wake Events service are optional and can be turned ON/OFF in kconfig.h.</p>
</div>
</div>
<div class="sect2 data-line-463">
<h3 id="remarks_on_synch_services">4.5. Remarks on synch services</h3>
<div class="sect3 data-line-465">
<h4 id="about_event_flags_and_event_groups">4.5.1. About event flags and event groups</h4>
<div class="paragraph data-line-466">
<p>Sometimes we are interested in the <strong>combination</strong> of the absence/presence of several events on a given interval - so, we need a storage. RTOSes often provide a service for that: 'event flags' within 'event groups'.  The subscribers sleep after a <code>get()</code> with the values they expect. The publisher <code>set()</code>. The broker reads, evaluates and <code>wake()</code> all interested readers.
As seen, it describes a public mailbox storing an integer (with flags). You will need more data structures, usually as simple as a table. But how the execution units handle this data  - on my perspective - does not justify new abstractions. It means more code, bugs, maintenance, and another API the kernel designer makes clear—another the user needs to learn.
(Hoare usually discussed the delicate balance that exists between specialisation and generalisation. Hansen would argue that opaque complexity within a concurrent environment will worsen things.)</p>
</div>
</div>
</div>
<div class="sect2 data-line-470">
<h3 id="about_blocking_primitives_time_out">4.6. About blocking primitives time-out</h3>
<div class="paragraph data-line-472">
<p>It is common for RTOSes to have a time-out the user assigns when calling a sleeping primitive. I wonder the right way to use this feature. What does a task do after this time-out? If a task tries to access a critical region and blocks forever, it is probably a system failure. We are better off catching it and fixing it up. As I see, the best approach would be to sleep on a timer (<code>sleep(ticks)</code>), wake up and test if the condition is met - if not, branch for a clean-up/recovery or even a reset. You might also test the condition before locking. Anyway, for semaphores, this feature is available - you use the <code>attempt (s, n)</code> primitive, providing the number of TICKS before giving up trying to acquire a semaphore. In this case the task will enter a special state called <code>ATTEMPTING</code>.
After N ticks have elapsed, if it can&#8217;t acquire the resource/region, the call returns an error code. The user might check for this return so the task can branch to an alternative path. You can use a <code>query()</code> to get the current value of the counter semaphore before waiting. For mutexes, you can query its lock state; for events, you can query the number of tasks within its sleeping queue.
<strong>Finally</strong>, for this usage pattern—checking before blocking—the tailored primitive within K0BA is an asynchronous mailbox: there is no kernel-imposed context-switching for a retry.</p>
</div>
</div>
<div class="sect2 data-line-476">
<h3 id="about_signals_installed_callbacks">4.7. About "signals" installed callbacks</h3>
<div class="paragraph data-line-478">
<p>It is tempting to become UNIX-minded and install callbacks on signals. When resuming, a task first checks for any pending signals. If they exist, the task deviates from its normal flow (as within any software interrupt), runs the callback, and deals with any side effects.
The overhead is excessive—both in time and space—a need to reserve a space on the stack or provide a stack for the callback; a need to tune the context-switching to happen when there is a signal pending and to get back to the deviated point. Since it kills any determinism, it is out of the kernel&#8217;s concerns—despite real-time  gospel says they are a big no; as I am not up for any gospel, I&#8217;m just providing a guideline. The application timers have installed callbacks (as they are periodic, the kernel can handle them).</p>
</div>
</div>
</div>
</div>
<div class="sect1 data-line-481">
<h2 id="inter_task_communication_services">5. Inter-task communication services</h2>
<div class="sectionbody">
<div class="sect2 data-line-482">
<h3 id="indirect_mailboxes">5.1. Indirect Mailboxes</h3>
<div class="paragraph data-line-483">
<p>In K0BA, a mailbox is a kernel object whose capacity is 'one message'. You write to a mailbox, then it is full. You read, it is empty. A mailbox stores a 32-bit address. The difference between a mailbox and a regular message queue (and these terms are often interchangeable) is that a message queue can be depicted as a circular data structure with a mailbox on the head and another on the tail. When both are full/empty, the queue will be logically full/empty.</p>
</div>
</div>
<div class="sect2 data-line-485">
<h3 id="blockingsynchronous_mailboxes">5.2. Blocking/synchronous mailboxes</h3>
<div class="paragraph data-line-487">
<p>These mailboxes use the primitives <code>post()</code> to write on a mailbox, and <code>pend()</code> to read from a mailbox. A <code>post()</code> returns a code (successful or a specific error) and a <code>pend()</code> returns the task ID of the sender - if successful.</p>
</div>
<div class="paragraph data-line-489">
<p>There is no copy within the mechanism - the message address is exchanged. If the consumer needs to copy, and message size is unknown, it can use a <code>mboxgetsize()</code> to know the size of the message. This size has to be declared when the mailbox is initialised, but is <strong>not mandatory</strong> (the kernel does not check and does not limit this value). A mailbox can be initialised full, if passing a valid initial mail pointer.</p>
</div>
<div class="paragraph data-line-491">
<p>When tasks block on a synch mailbox (posting on a full, reading from an empty), they are released (gain access) <em>either</em> by priority <em>or</em> on a first-in-first-out discipline (that is configured in compile-time <code>kconfig.h</code>).</p>
</div>
<div class="paragraph data-line-493">
<p><em>Two</em> tasks can fully synchronise on a command-response pattern, if a writer issues a <code>post()</code>  followed by a <code>pend()</code>; the reader issues a <code>pend()</code>  followed by a <code>post()</code>. If, a producer-only is not interested on a response, it might just post again, eventually pending on a full mailbox. After a task reads, it get access and deposits its message again.</p>
</div>
<div class="paragraph data-line-495">
<p>Alternatively (and arguably), two tasks might choose to go in lock-step skipping a response message, if a producer <code>pend()</code> after successfuly posting a message and a receiver <code>signal(id)</code> after receiving a message. This might be useful if generating and transmitting more messages is conditioned to having the previous message received. It is arguable because in this case an asynchronous mailbox wrapped by a user-defined synch protocol is probably better.</p>
</div>
<div class="paragraph data-line-497">
<p>A 'side-effect-feature', synch mailboxes are binary semaphores in their own right—they initialise with a state (full/empty), have a built-in priority inheritance mechanism and have a waiting queue - and they do not depend on the semaphore service.</p>
</div>
</div>
<div class="sect2 data-line-499">
<h3 id="asynchronous_mailboxes">5.3. Asynchronous mailboxes</h3>
<div class="paragraph data-line-501">
<p>Asynch mailboxes don&#8217;t have built-in blocking mechanisms and rely on
the value of the mail address attached to them: if NULL, it is empty.
A regular send() primitive to write on an async mbox will only be successful
if the box is empty (NULL mail) and a <code>recv()</code>, will copy the contents of
the mail address to a buffer the consumer passed as a parameter. The mailbox
will 'destroy' the message, so it is empty again.
A <code>recvkeep()</code> primitive does not destroy the message, and a <code>sendovw()</code> does not
  care if the box is full or not - it overwrites.</p>
</div>
<div class="paragraph data-line-510">
<p>When combined with synchronisation services, you can create flexible
communication channels.</p>
</div>
<div class="paragraph data-line-513">
<p>For instance, you can create a BROADCAST communication channel. Say,
an ISR deposits a message from a peripheral on an asynch mbox, and wakes
all tasks sleeping on an event associated to the mailbox and returns.
The woken tasks use <code>recvkeep()</code> - thus every task will get a copy of the message. (Within an ISR we cannot use blocking primitives).</p>
</div>
<div class="paragraph data-line-518">
<p>I regard mailboxes as powerful building blocks for task management, particularly in their asynchronous and synchronous extremes.</p>
</div>
<div class="paragraph data-line-520">
<p>Both synch mailboxes and asynch mailboxes are optional and can be turned ON/OFF in kconfig.h.</p>
</div>
</div>
<div class="sect2 data-line-522">
<h3 id="indirect_blocking_message_queue">5.4. Indirect Blocking Message Queue</h3>
<div class="paragraph data-line-524">
<p>The message queue service can carry messages up to 255 bytes. It works with an allocator. Given the circular FIFO nature, a <code>recv()</code> primitive copies the message to an address the receiver provides, so the queue buffer can be reused without corrupting data.</p>
</div>
<div class="paragraph data-line-526">
<p>You can choose to not copy messages but send and receive pointers. In this case, the memory you statically allocated for the queue is not passed as a parameter during the initialisation, you pass it as <code>NULL</code>, and use the <code>sendptr()</code> and <code>recvptr()</code>. The way you manage you memory is up to you, but, I recommend using the allocator service. You send the message address and guarantees it will be in scope when the receiver gets it - the message queue service will not touch this pool. The receiver might it self after calling `recvptr()`free the memory.
 This method will work particularly when you do not need a <em>deepy</em> copy, that is, the data does not points to other pointers.</p>
</div>
<div class="paragraph data-line-529">
<p>A message queue will behave asynchronously until a receiver try to read from an empty queue, or a producer tries to write on a full queue. So, they will block. This behaviour might be desirable or not. The result, anyway, is that tasks with different periods end up synchronising to the lowest rate task -  (<em>usually</em>, undesirable for real-time). After a queue blocks full, when it eventually unblocks you might had lost data. Is it the problem? Yes and no. The problem with a queue is that by buffering you are reading from the past. Between a message that does not arrive, and one that  arrives with information that is not useful, there is no practical difference. But if a message arrives with information that will deceive your control loop to perform a wrong action - this is the worse.  That can be avoided if your send-receive pair follows an unidirectional  order, and handle predictable timing events (loosely, periodic events). Finally, the queue size must be adequate (look at Little&#8217;s theorem).  So, except for logging/monitoring or serialize access, queues <em>might</em> cause trouble when a system takes action with the information it is reading.</p>
</div>
<div class="paragraph data-line-531">
<p>Important to note that although the Message Queue API asks for the address for the message contents, there is an internal pool of 'messages records': they wrap the message address with the sender ID and the message size, and that is what is transmitted/received. Then, a message queue internally is a list of message records (&#8230;&#8203; this was done so you can send messages <em>up to</em> a maximum size within the same queue, and identifying the sender ID is often useful). The number of message records is defined in <code>kconfig.h</code>
Also, you can make the <code>send()</code> primitive non-blocking, so it will overwrite the current message on the tail if the queue is full (<code>kconfig.h</code>).
Blocking <code>send()</code> and <code>recv()</code> releases blocked tasks by priority.</p>
</div>
<div class="paragraph data-line-535">
<p>(I personally rarely use service queues when working with 3rd party rtoses. A linear/circular buffer or a list + semaphores will do the job with no overkill - if it is shared memory. Consider that before using this service.)</p>
</div>
<div class="exampleblock data-line-539">
<div class="content">
<div class="paragraph data-line-540">
<p>In UNIX a broken pipe is a pipe with writers but no
readers. A pipe with readers but no writers is not 'broken'. It reinforces the idea that there is not such a thing as valid never-received data.</p>
</div>
</div>
</div>
<div class="paragraph data-line-544">
<p>Message queues are optional and can be turned ON/OFF in <code>kconfig.h</code>.
You cannot receive from a queue within ISRs. To send from an ISR, it shall be non-blocking.</p>
</div>
</div>
<div class="sect2 data-line-547">
<h3 id="asynch_pump_drop_message_queue">5.5. Asynch Pump-Drop Message Queue</h3>
<div class="paragraph data-line-548">
<p>These are queues meant to overcome the drawbacks exposed above for message queues, based on the concept of Cyclical Asynchronous
  Buffers (CAB) (essentialy as found in the HARTIK kernel). These queues are a one-to-many mechanism. A buffer can be read by several consumers, but no buffer can be written while consumed. The ideal queue size is the number of readers + 1.</p>
</div>
<div class="paragraph data-line-551">
<p>The semantics is as follows:</p>
</div>
<div class="ulist data-line-553">
<ul>
<li class="data-line-553">
<p>When the producer needs to write a new message to the queue, first it reserves a buffer. With a valid address, it attaches to the buffer the mesage address and size, and 'pump' it to the queue. Now, this message is available to be read by several consumers - until it is overwritten.</p>
</li>
<li class="data-line-554">
<p>The last message pumped in a queue is not overwritten UNTIL a new buffer is
pumped and finds its number of clients is zero (all readers dropped)</p>
</li>
<li class="data-line-556">
<p>A reader first 'fetches' a message buffer address - what will increase the user count within a buffer. After consuming the message, the reader 'drop the buffer'.</p>
</li>
</ul>
</div>
<div class="paragraph data-line-558">
<p>By the exposure, you can conclude that a reader might miss messages when the writer&#8217;s rate is higher. A reader within a (servo-loop) controller can afford a 'miss' rate— what leads to failure is reading messages that do not represent the plant state.</p>
</div>
<div class="paragraph data-line-560">
<p>Pump-drop queues are optional and can be turned ON/OFF in kconfig.h.</p>
</div>
</div>
<div class="sect2 data-line-562">
<h3 id="pipes">5.6. Pipes</h3>
<div class="paragraph data-line-564">
<p>Pipes transmit streams of bytes - unlike queues or mailboxes
that transmit 'objects' with a format. They are fast because data does
not move, what moves is the read/write pointer within the pipe buffer.
A writer stops its operation when the N specified bytes have been
V or there is no more room. In the last case it goes to sleep until
a reader wakes it up after 'extracting' (moving the read pointer) from the
  pipe.</p>
</div>
<div class="paragraph data-line-572">
<p>A reader reads until N specified bytes were read or when there is no more data. In the last case it will sleep, to be waken by a writer.
Both one-to-many and many-to-one channels are possible - those who wake need to themselves ensure precedence.</p>
</div>
<div class="paragraph data-line-575">
<p>Pipes are optional and can be turned ON/OFF in kconfig.h. You cannot usem them within ISRs, in no occasion.</p>
</div>
